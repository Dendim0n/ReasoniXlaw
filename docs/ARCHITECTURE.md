# Architecture

## Overview

This is a **ContextEngine plugin** for OpenClaw that provides prefix-stable
context management for DeepSeek-compatible target models.

It does NOT replace PI's execution loop. It plugs into the ContextEngine
interface at two points: `assemble()` and `compact()`.

## Runtime Contract

| Contract | Current value | Notes |
|----------|---------------|-------|
| Plugin entry id | `reasonixlaw` | Use this under `plugins.entries` for new installs |
| Runtime context engine id | `reasonixlaw-prefix-stable` | Use this under `plugins.slots.contextEngine` |
| Legacy config key | `deepseek-harness` | Read only as a fallback for `config`, not the preferred entry id |
| Archive directory | `~/.openclaw/reasonixlaw/archive/` | New archive writes use the project id |
| State sidecar suffix | `.reasonixlaw-state.json` | Written next to the OpenClaw session file |
| Legacy sidecar suffix | `.deepseek-harness-state.json` | Read fallback for sessions created before the rename |

The public project name is ReasoniXlaw. Some internal symbols still use
`DeepSeekContextEngine` because the implementation began as a DeepSeek-specific
harness. `deepseek-harness` remains only for legacy config and legacy sidecar
restore. The runtime ids above are the user-facing contract.

## Integration Flow

```
User sends message
  │
  ├─ OpenClaw Gateway receives message
  │
  ├─ PI agent loop starts
  │   ├─ ContextEngine.assemble(messages, tokenBudget)
  │   │   └─ THIS PLUGIN: prefix-stable layer management
  │   │       ├─ Layer 1: LOCKED PREFIX (system + early history)
  │   │       ├─ Layer 2: ACTIVE TAIL (recent messages, append-only)
  │   │       └─ Layer 3: COMPRESSED MIDDLE (summary, only layer that changes)
  │   │
  │   ├─ PI calls model with assembled prompt
  │   │   └─ DeepSeek API sees stable prefix → cache hit → 90% savings
  │   │
  │   ├─ PI executes tool calls (handled by PI, not us)
  │   │
  │   ├─ PI appends results to transcript
  │   │
  │   └─ ContextEngine.compact() if context near full
  │       └─ THIS PLUGIN: prefix-stable compaction
  │           ├─ Summarize middle messages using runtime LLM
  │           ├─ Keep prefix + recent tail unchanged
  │           └─ Archive dropped messages to disk
  │
  └─ PI streams response to channel
```

## ContextEngine Methods Used

| Method | When Called | What We Do |
|--------|-----------|------------|
| `bootstrap()` | Session init | Store session ID, set up archive dir, restore in-memory or sidecar state |
| `assemble()` | Before every model call | Build prefix+summary+tail, diff new messages |
| `compact()` | When context nears limit | Token-budget tail selection, structured middle summary, stuck-loop guard |
| `maintain()` | After turns | No-op (future: periodic cleanup) |
| `ingest()` | Per-message tracking | No-op (assemble handles diffing) |
| `dispose()` | Plugin unload | Save in-memory and sidecar state |

## Compaction Strategy

```
Before: [PREFIX][msg1, msg2, msg3, msg4, msg5, msg6, msg7, msg8]
After:  [PREFIX][summary: "msg1-5 discussed X..."][msg6, msg7, msg8]
         ↑ NEVER CHANGES
```

Rules:
1. Never split tool-call/result pairs at the boundary
2. Archive dropped messages to `~/.openclaw/reasonixlaw/archive/`
3. Select the recent tail by token budget while preserving a minimum recent count
4. Summary generated by runtime LLM (fallback: extractive summary)
5. Recompress accumulated summary when it grows beyond `maxSummaryTokens`
6. Trim older tool results with an explicit marker, leaving recent tool results intact
7. Pause automatic compaction after repeated compactions fail to get below threshold
8. Prefix messages are locked once and not re-derived during later compactions

Layer state is also written to a small sidecar next to the OpenClaw session file:

```
<sessionFile>.reasonixlaw-state.json
```

The sidecar contains only the context-engine projection state: prefix, tail,
summary, ingest counters, last model, and compaction statistics. It lets the
engine recover prefix-stable layers after process restarts when the host supplies
a stable session file path. On bootstrap, the engine first tries the new
ReasoniXlaw sidecar and then falls back to the legacy
`<sessionFile>.deepseek-harness-state.json` file.

## Configuration Surface

| Option | Default | Effect |
|--------|---------|--------|
| `prefixLockCount` | 2 | Locks the earliest messages into the stable prefix |
| `recentKeepCount` | 8 | Always preserves at least this many recent messages |
| `tailTokenBudget` | 16384 | Keeps extra recent messages while they fit the budget |
| `compactRatio` | 0.8 | Starts compaction pressure near this context ratio |
| `outputReservedTokens` | 32768 | Reserves room for model output before calculating pressure |
| `maxSummaryTokens` | 2000 | Recompresses accumulated summaries above this estimate |
| `toolResultTrimChars` | 2000 | Trims older tool results to this many characters |
| `archiveDropped` | true | Writes compacted-away messages to local JSONL archive |
| `targetModels` | DeepSeek V4 + Mimo aliases | Adds model patterns beyond the built-in `deepseek` substring rule |

The manifest declares the same config keys in `openclaw.plugin.json`. This
matters because `additionalProperties: false` would otherwise reject documented
knobs that are not listed in the schema.

## Observability

`getCacheStats()` reports the current layer shape:

- `prefixTokens`, `summaryTokens`, `tailTokens`, `totalTokens`
- `cacheHitEstimate`, calculated as `prefixTokens / totalTokens`
- `prefixMessages`, `tailMessages`
- `compactionCount`
- `consecutiveOverThresholdCompactions`
- `compactStuck`
- `lastCompactionTokensBefore`, `lastCompactionTokensAfter`

These numbers are estimates. They use the local CJK-aware token estimator in
`src/types.ts`, not the provider tokenizer. Treat them as operational signals,
not billing-grade accounting.

## Tradeoffs

Benefits:

- Lower DeepSeek input cost when the locked prefix hits provider cache.
- Higher cache hit probability because compaction does not rewrite the prefix.
- Better control over long sessions through token-aware tail selection, bounded
  summaries, old tool-result trimming, and stuck-loop detection.
- PI still owns auth, tools, retries, streaming, transcript persistence, and
  provider cache telemetry.

Costs:

- Early messages become sticky. If the session starts with bad instructions or
  stale assumptions, those messages stay in the prefix until a new session.
- Summaries are lossy even when structured. The prompt preserves state, files,
  commands, errors, and next steps, but exact detail can still be dropped.
- Runtime LLM summarization adds latency and token cost during compaction. The
  fallback is cheaper but less precise.
- Tool-result trimming can hide old evidence from the model. The trim marker
  records the original and retained lengths, and archive files can preserve the
  original locally.
- Sidecar and archive files retain local context data. That is useful for
  recovery and audit, but it is still local persistence.
- When `compactStuck` is true, automatic compaction pauses. This avoids repeated
  waste, but the host may need to apply its own pressure handling.

## Why Not AgentHarness?

AgentHarness replaces the **entire** execution loop (Codex uses this because
it has its own app-server). We'd have to reimplement:
- Tool execution
- Auth & API key resolution
- Streaming
- Retries & fallback
- Transcript persistence

ContextEngine gives us exactly what we need (prompt assembly + compaction)
while reusing PI's battle-tested execution infrastructure.
